## 教材へのアクセス方法（重要・再開時に使う）

ZEN Study（旧N高，`nnn.ed.nico`）の教材は要ログイン。
agent-browser の保存セッション `nnn` にログイン状態が保存済み（公共哲学と共通）。

### ⚠️ 公共哲学との違い：解析学はスライド画像ではなくテキスト

公共哲学はスライドPNGをDLしてOCR的に読む方式だったが、
**解析学1の教材は HTML のテキスト本文 ＋ MathJax(v2) の LaTeX 数式**。
テキストとして綺麗に抽出できる（画像DL不要）。

数式は `script[type="math/tex"]` に元の LaTeX が入っているので、そこから抜くと
MathJax の3重重複（`y=f(x)y=f(x)y=f(x)` のような表示）を避けてクリーンに取れる。

### 取得手順

```bash
# 抽出器（div.section ごとに本文を取り、数式を $LaTeX$ に置換）
#   → scratchpad/extract.js（下記）
# ラッパー：URLを開いて3秒待って extract.js を eval
#   → scratchpad/fetch_text.sh

agent-browser --session-name nnn open "<教材URL>"
agent-browser --session-name nnn wait 3000
agent-browser --session-name nnn eval "$(cat extract.js)"
```

**既知の小バグ**：ディスプレイ数式（`$$...$$`）の直前に、レンダリング済みテキストが
ゴミとして1行漏れることがある（例 `Γ(x)=∫∞0...` の行）。書き起こし時に手で除去する。
インライン数式（`$...$`）は問題なし。

🔴🔴**最重要バグ（2026-06-28 ユーザー指摘で発覚）：extract.js は `div.section` しか拾わず、
例題 `div.example`・練習問題 `div.problem` を丸ごと取りこぼす。**
- これらのボックスは `div.section` の外（body直下）にある。
- 番号「問題2-2-1」「例題2-3-1」は CSS `::before{content:"問題" attr(data-zen-number)}` で生成され、
  **DOMテキストに存在しない**（テキスト検索では絶対に見つからない）。**番号は `data-zen-number` 属性から取る。**
- 解答は `<details>`＋「解答を表示」ボタンで隠れている。抽出前に `details` を open し、ボタンを click する。
- → **修正版抽出器 = scratchpad/extract3.js**（section + 全ボックス種を出現順に拾い、番号は属性から）。再生成は STUDY_LOG 末尾参照。
- **教訓：各セクションを取得したら必ず「example/problem ボックスが無いか」を data-zen-number で確認する。演習を落とすと学習の核心を落とす。**

### extract.js の中身（再生成用）

```javascript
(() => {
  function sectionToMd(sec) {
    const clone = sec.cloneNode(true);
    const scripts = Array.from(clone.querySelectorAll('script[type^="math/tex"]'));
    scripts.forEach(s => {
      const tex = s.textContent;
      const display = (s.getAttribute('type')||'').includes('mode=display');
      const wrap = display ? ' $$'+tex+'$$ ' : ' $'+tex+'$ ';
      const repl = document.createTextNode(wrap);
      let prev = s.previousSibling;
      while (prev && prev.nodeType === Node.ELEMENT_NODE && /MathJax/.test(prev.className||'')) {
        const toRemove = prev; prev = prev.previousSibling; toRemove.remove();
      }
      s.replaceWith(repl);
    });
    clone.querySelectorAll('script').forEach(n=>n.remove());
    let txt = clone.textContent.replace(/[ \t]+/g,' ');
    txt = txt.split('\n').map(l=>l.trim()).filter(l=>l.length).join('\n');
    return txt;
  }
  return JSON.stringify(Array.from(document.querySelectorAll('div.section')).map(sectionToMd));
})()
```

### extract3.js（★これを使う：section＋例題＋練習問題を全部拾う・番号は属性から）

数式置換ロジックは上と同じ。違いは「拾う対象」と「番号の取り方」。
**抽出前に必ず**：`document.querySelectorAll('details').forEach(d=>d.open=true)` と
「解答を表示」ボタンの click を実行（解答が隠れているため）。

```javascript
(() => {
  function md(sec){const c=sec.cloneNode(true);
    c.querySelectorAll('script[type^="math/tex"]').forEach(s=>{const t=s.textContent;
      const d=(s.getAttribute('type')||'').includes('mode=display');
      const r=document.createTextNode(d?' $$'+t+'$$ ':' $'+t+'$ ');
      let p=s.previousSibling;while(p&&p.nodeType===1&&/MathJax/.test(p.className||'')){const x=p;p=p.previousSibling;x.remove();}
      s.replaceWith(r);});
    c.querySelectorAll('script').forEach(n=>n.remove());
    return c.textContent.replace(/[ \t]+/g,' ').split('\n').map(l=>l.trim()).filter(l=>l).join('\n');}
  const sel='div.section, div.intro, div.summary, [data-zen-number], div.problem, div.example, div.definition, div.theorem, div.proposition, div.proof, div.hint, div.answer, div.note';
  const seen=new Set(), out=[];
  document.querySelectorAll(sel).forEach(n=>{
    if(seen.has(n))return;
    let p=n.parentElement,nested=false;
    while(p){if(p.matches&&p.matches(sel)){nested=true;break;}p=p.parentElement;}
    if(nested)return; seen.add(n);
    const cls=(n.className||'').toString(), kind=cls.split(/\s+/)[0]||n.tagName.toLowerCase();
    out.push({kind, num:n.getAttribute('data-zen-number')||'', md:md(n)});
  });
  return JSON.stringify(out);
})()
```

## 学習設計の根拠（認知科学・数学教育リサーチの結論）

過去の挫折（線形代数2・距離空間・集合と論理で証明に力尽きた）の正体と対策を、
研究ベースで設計した。**挫折は能力でなく「読み方と負荷管理」の問題で、設計で解ける。**

### 過去の挫折の正体（本人の証言）
1. **前提知識の欠落**（証明が暗黙に使う過去の定理・定義が曖昧）
2. **行間の飛躍**（1行目と2行目がなぜ繋がるか分からない）
3. → **それが続いて精神的に辛くなる**（最大の要因）

### 効く対策（→ 出典は本セクション末尾）

| 過去の失敗 | 研究ベースの対策 |
|---|---|
| 行間が埋まらない | **自己説明(self-explanation)**：各行で「直前のどの情報＋既知のどの定理から従うか」を1問だけ自問。言い換え(NG)・実況(不十分)と区別。効果量 d≈0.95 |
| 全部理解しようとして力尽きる | **証明は最低2パス**。1パス目は型と骨格だけ（理解を意図的に保留）。2パス目で引っかかった行だけ埋める。**大域→局所→例**の順 |
| 負荷で過負荷 | **ワークト例→フェーディング**。新証明はまず完成証明を例として読む。慣れたら最後のステップから隠して自力化（逆向きフェーディング） |
| 精神的に辛い | 「辛い＝能力不足」ではなく「**不安がワーキングメモリを食っている**」と解釈し直す。自己批判しない（セルフコンパッション）。**15分ルール**で生産的/破壊的苦闘を判定 |
| 定着しない | **検索練習**（学んだ証明を後日、何も見ずに骨格から再構成）＋**分散**（短いセッションを数日おき）＋**インターリービング**（手法をある程度学んでから混ぜる）。再読・ハイライトは最低ランク |
| AIで分かった気になる | AIに**答えを出させない**（ヒント1歩ずつ）。新定理はまず自力で予想してから足場を求める（生産的失敗）。AIの主用途は**前提欠落の診断** |

### 1セッションの運用テンプレート（気力が有限な前提）
1. （任意）開始1〜2分、不安・懸念を書き出す（表現的筆記でWM確保）
2. 新しい証明：まず型と骨格だけ1パス（理解を保留）→ 完成証明を例として各行を自己説明で2パス目 → **具体例を1つ当てる**
3. 詰まったら15分ルールで判定。破壊的なら前提を最小限で埋めるか、自分の仮説を先に言ってAIに足場を求める
4. 慣れた証明は最後のステップから隠して自力補完（逆向きフェーディング）
5. 後日（数日おき）：学んだ証明を何も見ずに骨格から再構成（spaced retrieval）。たまったら手法を交互に混ぜる

## 🔴 最重要：質問に「間違った前提」を埋め込まない（再発防止プロトコル）

過去に何度も、AIがソクラテス式の質問の中に**誤った前提を埋め込み**、生徒の限られた気力を浪費させた。
例：「求めたいのは eˣ」（本当は eˣ を近似する一次関数）／問題が要求していない「h を x に直せ」を勝手に課す／
前の問題の設定（中心 a、文字 h）を別の小問に引き継いで間違える。**毎回 生徒が正しく検知して止めた。**

### なぜ「自分で前提を確認する」だけでは直らないか（研究の結論）
- LLM は外部フィードバック無しの自己修正が苦手で、むしろ性能が下がることがある（Huang 2024, ICLR）。
- モデルは前提に逆らうより乗っかるよう偏っている＝前の問題の誤前提がこびりつく（Sharma 2023, Anthropic, sycophancy）。
- → **内省の自己チェックは信頼できない。確認役は生徒（外部チェッカー）に置く。** だから前提を「生徒に見える形」で外に出す。
- 公式ドキュメントも「作業前に逐語で引用させると回答が実テキストに接地して幻覚が減る」と明記（Anthropic, Reduce hallucinations）。

### 守ること（命綱）
1. **問題文は逐語で扱う。** 自作問題を出すときは「これは私の自作問題です」と明示する。教材の問題は原文のまま貼る。言い換えない。
2. **「いま求めるもの」は、その問題文に書いてあることだけから読む。** 一般知識・前の問題・自分の予想で補完しない。

3. **前の問題の設定（中心 a・文字 h など）を別の小問に引き継ぐ前に、問題文を読み直してゴールをゼロから導出し直す**（前の結論を一旦無視する）。
4. **質問は1ターンにつき1つだけ。** 長い表・複数レイヤーの説明を一度に出さない（量が増えると前提の歪みも増える）。

これは完璧な保証ではない。AIが もっともらしく歪んだ「求めるもの」を書く可能性は残る。

## 🔴 出力テンプレート：1ターン＝1要素だけ出す（先回り・冗長の物理的封じ込め・2026-06-30 全面改訂）

### なぜ「〜するな」では直らないか（研究で裏取り済み・出典は下記）
過去に何度も「ヒントを先回りするな」「短く」「次の問いを出すな」と**禁止形**で指示したが、
**追記した直後にまた再発した**（本セッションの transcript で実証）。理由は3つ：
1. **Verbosity / over-helpfulness bias**：RLHF で「詳しく親切な応答」が報酬で強化されている。
   だから推論時の「短く」より、訓練済みの「もっと説明したい」が勝つ。
2. **負の指示は効かない（Pink Elephant 問題）**：「次の問いを出すな」と言うと、AI は「次の問い」を
   想起した時点で出してしまう。LLM は否定を直接処理できない。
3. **Context rot**：コンテキストが長くなると初期の制約への注意が薄れる（だから最重要(0)＝1節で /clear）。
→ 対策：**禁止形をやめ、「何を出すか」を実行形のテンプレートで固定する。** これが本節。

### 守ること＝下の「出力テンプレート」に必ず当てはめる（実行形）

**生徒が問題を解いて答えを出したとき（答え合わせ）→ 出すのは次の2つだけ：**
```
[正誤と、合っている／違う箇所の一言。最大2〜3文]
[次の一手をこちらから足さない。生徒の次の入力を待つ。]
```
- 別解・作法（「(1+x²)²のまま残すのが普通」等）・メタ学習論（「これが検索練習」等）は**足さない**。
- 「次は2-6-X をやりますか／一区切りにしますか」の**選択肢提示もしない**。生徒が次を自分で指定する。
  進行の決定権は生徒なので、生徒が「次どうする?」と聞いてきたときだけ選択肢を出す。

**生徒が新しい問題に入るとき → 出すのは次の1つだけ：**
```
[問題文（教材原文 or 自作明示）を1つ。それだけ。]
```
- 部品整理（g=…, f=…）・公式の提示・「まず○○は何?」という誘導の問い＝**出さない**。生徒がゼロから書く。

**生徒が「詰まった」と言った／ヒントを要求したとき → 出すのは次の1つだけ：**
```
[最小の足場を1つ。方向づけ1文 or 未習事項1行のみ。]
```
- 足場に**部品整理や次の問いを付けない**。1要素出したら止まる。答えそのものは最後まで出さない。

**生徒が質問したとき → 聞かれたことだけに答える。** 用途・比喩・関連トピックへの拡張は、生徒が
「もっと」と言ったときだけ。前置き（「すごく良い質問です」等）は1文まで。

### 自己チェック（応答を送る前に必ず1回）
送信前に「**この応答に、生徒が今のターンで求めていない要素が混じっていないか?**」を1回確認する。
混じっていたら削る。特に頻発する4つ：①答え合わせ後の「次どうする」選択肢 ②正解への別解・作法・意義づけ
③問題提示時の部品整理・誘導の問い ④足場1行のはずが部品整理＋次の問いまで。**これらは全部削る。**

### 大前提（変わらない学習設計）
